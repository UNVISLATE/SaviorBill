# SaviorBill биллинг

**SaviorBill** — современная, высокопроизводительная биллинговая система, спроектированная как надежная и гибкая альтернатива устаревшим решениям. Проект построен на принципах реактивности и максимальной изоляции логики интеграций.

## Философия проекта
SaviorBill создается с нуля для решения проблем производительности и поддержки, характерных для систем прошлого поколения. Основной фокус — стабильность расчетов, прозрачность событий и возможность обновления интеграций без остановки ядра системы.

## Технологический стек
- **Ядро (Core):** `Python` (FastAPI + SQLAlchemy) — управление бизнес-логикой, балансами и состоянием системы.
- **Движок интеграций:** `Lua` — выполнение внешних взаимодействий (платежные системы, API-хуки) в изолированной среде.
- **База данных:** `PostgreSQL`.
- **Кэширование и очередь задач:** `Valkey` (Форк Redis) — для хранения временных данных и управления событиями.
- **Архитектура:** `Event-Driven` + `Time-Event Driven` (реактивное управление состоянием).

## Ключевые архитектурные решения
1. **Разделение ответственности:** Python отвечает за "деньги и логику", Lua — за "взаимодействие с миром". Это позволяет обновлять платежные модули "на лету".
2. **Реактивность:** Система не опрашивает базу данных в циклах (polling). Все изменения (оплата, списание, продление) происходят через события.
3. **Оптимизация нагрузки:** Использование временных бакетов для обработки массовых задач и встроенный `Grace Period` для сглаживания пиковых нагрузок на БД в моменты изменения/отключения/продления услуг.
4. **Единообразие:** Весь код придерживается строгих стандартов форматирования (`Black`) и типизации, что обеспечивает чистоту и читаемость даже при сложной бизнес-логике.
5. **Система ролей и прав доступа:** Гибкая настройка прав для администраторов, операторов, партнеров и клиентов через централизованную систему авторизации.

## Почему это работает лучше?
- **Безопасность:** Отказ от небезопасных shell-скриптов и устаревших интерпретаторов в пользу изолированных сред.
- **Масштабируемость:** Событийная архитектура позволяет системе работать под высокой нагрузкой без бесконечных проверок состояния пользователей.
- **Гибкость:** Поддержка нового платежного шлюза добавляется через Lua-скрипт без необходимости пересборки основного ядра на Python.
