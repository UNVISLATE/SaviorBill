# Для разработчиков

Здесь описаны принципы работы некоторых систем и структура проекта, так как код на данный момент не полностью документирован.

## 1. Иерархическая система прав доступа (Hierarchical RBAC)
SaviorBill использует древовидную структуру прав. Права определяются как вложенные словари, что позволяет реализовать наследование доступа.

* **Принцип:** Если пользователю разрешен доступ к узлу `payment`, он автоматически получает доступ ко всем подпунктам (`create`, `refund`, `delete`).
* **Регистрация:** При старте приложения все эндпоинты сканируются и выстраиваются в дерево прав (`PermissionTree`).
* **Проверка:** `PermissionChecker` рекурсивно проверяет наличие пути в дереве прав пользователя.
* **Синтаксис:**
    * Router level: `permission="payment"`
    * Route level: `permission="refund"`
    * Результат: `payment.refund`

## 2. Система событий (Event-Driven Architecture)
Вся бизнес-логика построена на реактивном подходе. Мы избегаем `polling` (опроса БД) в пользу событийной модели.

*   **Event Bus:** Центральный узел для обмена событиями внутри приложения.
*   **Batch Processing:** Для массовых операций (например, продление подписок тысяч пользователей) запрещено создавать индивидуальные события. Используйте «бакеты» (buckets) — группировка пользователей по временным интервалам.
*   **Time-Event Driven:** Для задач по расписанию используйте планировщик как *Event Producer*. Он не выполняет логику, а лишь выбрасывает событие в шину.
*   **Grace Period:** Все события приостановки услуг должны учитывать `GRACE_PERIOD_MINUTES` из конфига, чтобы избежать ложных срабатываний и сгладить пики нагрузки на БД.

## 3. Стандарты интеграций (Lua Layer)
Lua используется исключительно как **Integration Engine**.

*   **Песочница:** Весь код на Lua изолирован от бизнес-логики ядра.
*   **Контракты:** Обмен данными между Python и Lua должен строго соответствовать Pydantic-схемам.
*   **Безопасность:** Lua-скрипты не имеют прямого доступа к БД, они взаимодействуют с системой через предписанный интерфейс (callback/event).

## 4. Аналитика (Analytics)
Используется Polars для аналитики и построения отчетов. Все события, которые могут быть полезны для анализа, должны логироваться в формате, пригодном для последующей агрегации.

## 5. Требования к коду (Code Style)
Для обеспечения единообразия соблюдайте следующие правила:

1.  **Форматирование:** Весь Python-код должен проходить через `black`.
2.  **Финансы:** Для всех валютных операций используйте тип `Decimal`. Категорически запрещено использование `float`.
3.  **Типизация:** Обязательное использование type hinting для всех публичных методов и параметров роутов.
4.  **Auto-Cleanup:** Для таблиц с логами или временными данными используйте `LimitTableMixin`, чтобы система была самоочищающейся.
5.  **Конфигурация:** Все параметры системы (периоды, лимиты, интервалы) выносятся в `.env`. Никаких "магических чисел" в коде.
