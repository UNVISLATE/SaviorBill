# Ротация ключа шифрования секретов (SecBox)

`SecBox` (`src/utils/sec/box.py`) шифрует секреты, хранимые в БД: цифровые
ключи товаров (`service_keys.value`), секретные системные настройки
(`settings.value` при `is_secret=true`), секреты платёжных провайдеров
(`payment_providers.secrets_enc`) и OAuth-провайдеров
(`oauth_providers.secrets_enc`).

## Формат `SECRETS_KEY`

Значение секрета `SECRETS_KEY` (резолвится через `utils/sec/secrets/`, см.
`resolve_secrets()`) — список ключей Fernet через запятую, с версионными
метками для читаемости: `"v2:<новый_ключ>,v1:<старый_ключ>"`. Порядок в
списке важен: **первый** элемент — текущий ключ для *записи* (`seal()`
всегда шифрует им), остальные — только для чтения уже существующих данных
при расшифровке (`open()` пробует все ключи по очереди через `MultiFernet`).

При первом запуске (когда `SECRETS_KEY` ещё не существует) секрет
генерируется автоматически уже в этом формате (`SecBox.new_versioned_key()`,
версия `v1`) — ротация никогда не требует миграции формата, только
добавления нового ключа первым в список.

## Процедура ротации

1. Сгенерировать новый ключ: `SecBox.new_key()` (или `python -c "from
   cryptography.fernet import Fernet; print(Fernet.generate_key().decode())"`).
2. Обновить секрет `SECRETS_KEY` в используемом хранилище (файл/AWS/GCP/
   Azure/Vault — см. `utils/sec/secrets/`), добавив новый ключ **первым**:
   `"v2:<новый>,v1:<старый>"`. Перезапустить (или дождаться штатного
   рестарта) сервис — с этого момента все *новые* записи шифруются новым
   ключом, а старые данные остаются читаемыми (`MultiFernet` пробует оба).
3. **Никакого немедленного bulk-rewrite не требуется** — старые
   зашифрованные строки продолжают открываться старым ключом сколь угодно
   долго, пока он в списке.
4. (Опционально, чтобы со временем убрать старый ключ) — запустить фоновую
   идемпотентную джобу ре-шифровки: `utils/sec/rotate.py::reencrypt_all(session,
   box)`. Проходит по всем известным зашифрованным колонкам, для каждой
   непустой строки делает `open()` → `seal()` (новым ключом) → commit
   **отдельной транзакцией на строку**. Падение процесса в середине прохода
   не портит данные: часть строк уже в новом формате, часть — в старом, обе
   версии читаемы, пока старый ключ ещё в списке. Джобу можно перезапускать
   в любой момент.
5. Когда джоба подтвердила (по логу `rotate: перешифровано строк: {...}` на
   повторном проходе — счётчики становятся малы/нулевые для уже
   обработанных строк, либо явной сверкой, что новых строк в старом формате
   не появляется), можно убрать старый ключ из `SECRETS_KEY`, оставив только
   новый: `"v2:<новый>"`. Строки, которые почему-то не попали под
   ре-шифровку, станут нечитаемы — поэтому шаг 4 обязателен перед этим.

## Юнит-тесты

`tests/unit/test_sec_box.py` покрывает: чтение старых данных новым
multi-key боксом, что `seal()` всегда использует первый ключ, CSV без
версионных меток, и что удаление старого ключа из списка закономерно рушит
расшифровку данных, зашифрованных только им (ожидаемое поведение — все
данные предварительно должны быть перешифрованы, см. процедуру выше).
