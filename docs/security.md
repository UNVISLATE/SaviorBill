# Ротация ключа шифрования секретов (SecBox)

`SecBox` (`src/security/sec/box.py`) шифрует секреты, хранимые в БД: цифровые
ключи товаров (`service_keys.value`), секретные системные настройки
(`settings.value` при `is_secret=true`), секреты платёжных провайдеров
(`payment_providers.secrets_enc`) и OAuth-провайдеров
(`oauth_providers.secrets_enc`).

## Формат `SECRETS_KEY`

Значение секрета `SECRETS_KEY` (резолвится через `security/sec/secrets/`, см.
`resolve_secrets()`) — список ключей Fernet через запятую, с версионными
метками для читаемости: `"v2:<новый_ключ>,v1:<старый_ключ>"`. Порядок в
списке важен: **первый** элемент — текущий ключ для *записи* (`seal()`
всегда шифрует им), остальные — только для чтения уже существующих данных
при расшифровке (`open()` пробует все ключи по очереди через `MultiFernet`).

При первом запуске (когда `SECRETS_KEY` ещё не существует) секрет
генерируется автоматически уже в этом формате (`SecBox.new_versioned_key()`,
версия `v1`) — ротация никогда не требует миграции формата, только
добавления нового ключа первым в список.

## Процедура ротации

1. Сгенерировать новый ключ: `SecBox.new_key()` (или `python -c "from
   cryptography.fernet import Fernet; print(Fernet.generate_key().decode())"`).
2. Обновить секрет `SECRETS_KEY` в используемом хранилище (файл/AWS/GCP/
   Azure/Vault — см. `security/sec/secrets/`), добавив новый ключ **первым**:
   `"v2:<новый>,v1:<старый>"`. Перезапустить (или дождаться штатного
   рестарта) сервис — с этого момента все *новые* записи шифруются новым
   ключом, а старые данные остаются читаемыми (`MultiFernet` пробует оба).
3. **Никакого немедленного bulk-rewrite не требуется** — старые
   зашифрованные строки продолжают открываться старым ключом сколь угодно
   долго, пока он в списке.
4. (Опционально, чтобы со временем убрать старый ключ) — запустить фоновую
   идемпотентную джобу ре-шифровки: `security/sec/rotate.py::reencrypt_all(session,
   box)` (либо через эндпоинт `POST /api/v1/admin/settings/secrets/reencrypt`,
   право `settings.secrets.reencrypt`). Проходит по всем известным зашифрованным колонкам, для каждой
   непустой строки делает `open()` → `seal()` (новым ключом) → commit
   **отдельной транзакцией на строку**. Падение процесса в середине прохода
   не портит данные: часть строк уже в новом формате, часть — в старом, обе
   версии читаемы, пока старый ключ ещё в списке. Джобу можно перезапускать
   в любой момент.
5. Когда джоба подтвердила (по логу `rotate: перешифровано строк: {...}` на
   повторном проходе — счётчики становятся малы/нулевые для уже
   обработанных строк, либо явной сверкой, что новых строк в старом формате
   не появляется), можно убрать старый ключ из `SECRETS_KEY`, оставив только
   новый: `"v2:<новый>"`. Строки, которые почему-то не попали под
   ре-шифровку, станут нечитаемы — поэтому шаг 4 обязателен перед этим.

## Юнит-тесты

`tests/unit/test_sec_box.py` покрывает: чтение старых данных новым
multi-key боксом, что `seal()` всегда использует первый ключ, CSV без
версионных меток, и что удаление старого ключа из списка закономерно рушит
расшифровку данных, зашифрованных только им (ожидаемое поведение — все
данные предварительно должны быть перешифрованы, см. процедуру выше).

## Подпись сообщений между сервисами (Valkey Streams)

`security/sec/bus_sign.py` — общий HMAC-SHA256 контракт для `lua/bus.py`
(`LuaBus`) и `messaging/mediabus.py` (`MediaBus`): подпись покрывает *все*
поля сообщения (не белый список, см. `docs/lua_scripts.md`/`main.lua`), с
окном `ts` против replay (`DEFAULT_MAX_SKEW_SEC` = 300с, константа модуля).
Секрет —
`BUS_SIGNING_KEY`, отдельный от `LUA_SERVICE_TOKEN` (у токена другая роль —
auth конкретного HTTP-вызова, не подпись очереди). Задачи (`*:tasks`) и
результаты (`*:results`) подписываются симметрично — закрывает и фальшивые
задачи, и фальшивые результаты воркера. Отказ проверки подписи не ретраится
через reclaim — сообщение подтверждается (`XACK`) как обработанное с
ошибкой, чтобы не зациклить ретраи заведомо поддельного сообщения; счётчик
отказов — метрика `bus_signature_rejected_total{bus}` (см.
`docs/telemetry.md`) — рост счётчика значит либо рассинхрон ключей на
сервисах, либо реальную попытку подмены сообщений.

## SSRF из Lua-скриптов (`httpc.lua`) — осознанно не ограничивается

`http{ url, ... }` в песочнице Lua-скриптов может обратиться куда угодно,
включая локальные адреса того же сервера — deny-лист/allowlist адресов не
реализован и **не планируется по умолчанию**. Решение принято осознанно:
оператор биллинга сам пишет/загружает Lua-скрипты (`POST
/api/v1/admin/lua/*`, право `lua.*` — уже привилегированное действие),
поэтому ограничение сетевого доступа скрипта от него же не защищает ни от
чего нового. Если понадобится в будущем (например, при допуске менее
доверенных авторов скриптов) — это опциональная per-instance настройка,
не переоткрывать как забытый баг в будущих аудитах.

## OAuth: подтверждение привязки к существующему аккаунту по email

Раньше первый вход через OAuth-провайдера, у которого верифицированный
email совпадал с уже существующим аккаунтом, приземлял пользователя в этот
аккаунт немедленно (`link_account` искал по email и просто логинил). Это
зависело от того, насколько строго конкретный провайдер проверяет владение
email — слабая проверка на его стороне = захват чужого аккаунта.

Теперь (`dependencies/oauth.py::OAuthSvc.link_account`) в этом случае
привязка не создаётся сразу:

1. Колбэк (`GET /api/v1/callback/oauth/{provider}`) возвращает
   `OAuthPendingLink{pending_token}` вместо `TokenPair`, и отправляет
   6-значный код на email **существующего** аккаунта (шаблон
   `oauth.link_confirm`, TTL — `mail.code_ttl`, как у сброса пароля).
2. Клиент подтверждает: `POST
   /api/v1/callback/oauth/pending/{pending_token}/confirm {code}` →
   создаётся привязка и выдаются токены для этого аккаунта.
3. Заявка и код хранятся только в Valkey (`oauth:pendinglink:*`), ничего не
   пишется в БД до подтверждения.

Путь `POST /api/v1/user/oauth/{provider}/link` (привязка ДОБАВЛЯЕТСЯ к уже
залогиненному пользователю) не менялся — там уже есть текущая сессия,
привязка добровольная, дополнительное подтверждение избыточно.

## Запрещённые для регистрации email-домены

Отдельная таблица `banned_email_domains` (`models/banned_email_domains.py`),
а не settings-ключ со списком — точный поиск по домену (PK, lowercase) на
каждую регистрацию, без парсинга JSON на каждый запрос.

Проверяется в двух местах создания нового локального аккаунта:

- `POST /api/v1/auth/register` (`api/v1/auth/local.py`) — до создания
  аккаунта, `400` при совпадении.
- `OAuthSvc.link_account` (`dependencies/oauth.py`) — при автосоздании
  нового аккаунта по OAuth-профилю (не при привязке к существующему), тот же
  `400`, чтобы обход через слабо проверяющий email OAuth-провайдер не
  позволял регистрацию с запрещённого домена.

Управление списком — `GET/POST /api/v1/admin/settings/email-domains`,
`DELETE /api/v1/admin/settings/email-domains/{domain}` (право
`settings.email_domains.read`/`.edit`; входит в общее `settings: true` роли
admin). Подгрузка публичных disposable-domain листов сознательно не
реализована — только ручное управление админом; можно добавить отдельной
задачей позже, если понадобится.

## Настройки: индексированный `prefix`

Таблица `settings` — `key`/`value`/`is_secret` (не менялось), плюс
вычисляемая СУБД колонка `prefix` (первый `.`-сегмент `key`, например
`smtp` для `smtp.host`; `GENERATED ALWAYS AS`, миграция
`0004_settings_prefix`). `SystemSettingsMngr.get_group()` ищет по ней
(индекс), а не `LIKE`/`startswith`-сканом всей таблицы.

Composite PK `(prefix, key)` сознательно не делался: слишком много мест
напрямую делают `session.get(SystemSettingsModel, full_key)` по полному
дотированному ключу (`api/v1/admin/settings/*.py`,
`security/sec/rotate.py`, `bootstrap/integrity_check.py`, тесты, сидинг) —
переход на композитный PK потребовал бы правки всех них при нулевой
функциональной выгоде (`get_group()` и так работает по единственному
реальному вызывающему — `dependencies/mail.py::"smtp."`). Статическая
группировка для админки/каталога настроек — отдельный механизм,
`SettingDef.group` (`core/settings_def.py`), не путать с этой колонкой.
